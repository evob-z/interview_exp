# 测试策略文档

> 版本：v1.2 | 最后更新：2026-05-26 | 状态：已更新（frontmatter_utils + mastery 测试覆盖）

## 1. 测试总体策略

### 1.1 测试金字塔

```
        /  E2E测试  \          ← 少量：全流程验证
       / 集成测试     \        ← 中量：API路由 + 模块协作
      / 单元测试        \      ← 大量：核心逻辑覆盖
     ──────────────────────
```

### 1.2 测试目标
- 核心业务逻辑覆盖率 ≥ 80%
- 所有API端点有冒烟测试
- LLM依赖全部Mock，测试可离线运行
- CI流水线 < 2分钟完成

### 1.3 测试工具栈
| 工具 | 用途 |
|------|------|
| pytest | 测试框架和断言 |
| pytest-asyncio | 异步测试支持 |
| pytest-mock (mocker) | Mock和Patch |
| FastAPI TestClient | API集成测试 |
| tmp_path fixture | 隔离文件系统测试 |

## 2. 测试分层

### 2.1 单元测试（`tests/unit/`）

**覆盖范围**：
| 模块 | 测试文件 | 关注点 |
|------|---------|--------|
| detector | `test_detector.py` | 文件分类、变更识别逻辑 |
| extractor | `test_extractor.py` | 格式检测、结构化解析 |
| archiver | `test_archiver.py` | 分类归档、去重检测 |
| frontmatter_utils | `test_frontmatter_utils.py` | YAML frontmatter 读写、wikilink 解析、inline 元数据 upsert |
| reflector (mastery) | `test_reflector.py` | mastery 打标、QuestionItem schema、enhance_mastery_from_summary |
| config | `test_config.py` | 配置加载、默认值 |
| profile_manager | `test_profile_manager.py` | 画像构建、增量更新 |
| prepare_agent | `test_prepare_agent.py` | Agent 模块导入、PrepareDeps 状态、submit_final 终结工具、fallback 逻辑、meta 透传 |

**单元测试原则**：
- 每个测试函数只测一个行为
- 测试命名：`test_<功能>_<场景>_<预期结果>`
- LLM调用全部通过 `mock_llm` fixture 替换
- 文件操作使用 `tmp_path` 隔离

**示例模式**：
```python
def test_detect_new_interview_file(isolated_repo):
    """新面试记录文件应被正确识别"""
    # Arrange
    (isolated_repo / "面试原始问题" / "字节_260513.md").write_text("Q1\nQ2")

    # Act
    result = detect_changes()

    # Assert
    assert len(result.new_files) == 1
    assert "字节_260513.md" in result.new_files[0]
```

### 2.2 集成测试（`tests/api/`）

**覆盖范围**：
| 端点 | 测试内容 |
|------|---------|
| POST /api/qa | 问答请求正常响应 |
| GET /api/profile | 画像数据返回格式 |
| POST /api/sync | 同步触发和结果 |
| GET /api/stats | 统计数据完整性 |
| GET /api/history/sessions | 会话列表格式 |

**集成测试原则**：
- 使用 FastAPI TestClient 发起真实HTTP请求
- Mock外部依赖（LLM、文件系统）
- 验证HTTP状态码 + 响应体结构
- 不验证LLM生成内容的具体文本

**示例模式**：
```python
def test_qa_endpoint_returns_answer(api_client, mock_llm):
    """问答端点应返回包含answer字段的响应"""
    mock_llm.set_response("这是一个测试回答")

    response = api_client.post("/api/qa", json={
        "question": "什么是RAG？",
        "mode": "quick"
    })

    assert response.status_code == 200
    assert "answer" in response.json()
```

## 3. Mock策略

### 3.1 LLM Mock（`mock_llm` fixture）

所有涉及LLM调用的测试必须使用Mock：
```python
@pytest.fixture
def mock_llm(mocker):
    """LLM调用桩，避免测试依赖真实API"""
    mock = mocker.patch("llm_client.chat_completion")
    mock.return_value = "默认Mock响应"
    return mock
```

**设计原因**：
- 测试不依赖网络和API Key
- 测试结果确定性（LLM输出不稳定）
- 测试速度快（无API延迟）
- CI环境无需配置密钥

### 3.2 文件系统隔离（`isolated_repo` fixture）

```python
@pytest.fixture
def isolated_repo(tmp_path, monkeypatch):
    """创建隔离的临时仓库结构"""
    # 创建目录结构
    (tmp_path / "面试原始问题").mkdir()
    (tmp_path / "面试复盘").mkdir()
    (tmp_path / "问题库").mkdir()

    # 重定向config路径
    monkeypatch.setattr("config.INTERVIEW_REPO_PATH", str(tmp_path))
    return tmp_path
```

### 3.3 API客户端（`api_client` fixture）

```python
@pytest.fixture
def api_client(mock_llm, isolated_repo):
    """FastAPI测试客户端，已Mock外部依赖"""
    from app import app
    return TestClient(app)
```

## 4. 测试执行

### 4.1 本地执行
```bash
cd AAA_manager

# 运行所有测试
pytest

# 运行单元测试
pytest tests/unit/

# 运行API测试
pytest tests/api/

# 运行单个测试文件
pytest tests/unit/test_detector.py

# 详细输出
pytest -v

# 显示print输出
pytest -s
```

### 4.2 pytest配置（pytest.ini）
```ini
[pytest]
testpaths = tests
pythonpath = .
filterwarnings =
    ignore::DeprecationWarning
asyncio_mode = auto
```

### 4.3 CI/CD集成
```yaml
# .github/workflows/test.yml
name: Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r AAA_manager/requirements.txt
      - run: cd AAA_manager && pytest --tb=short
```

## 5. 质量标准

### 5.1 通过标准
- 所有测试通过（零失败）
- 无未处理的Warning（已在pytest.ini中配置过滤）
- 核心模块（detector/extractor/archiver）测试覆盖核心路径

### 5.2 测试编写要求
- 新增功能必须同时提交对应测试
- Bug修复必须补充回归测试
- 测试代码同样遵循命名规范和注释规范

### 5.3 测试不覆盖的范围
- LLM生成内容的质量（通过人工评估）
- 前端UI交互（暂无前端测试框架）
- 第三方API的可用性（由Mock隔离）

## 6. 测试数据管理

### 6.1 Fixture文件
测试用的样本数据存放在 `tests/fixtures/`：
- `sample_interview.md`：非结构化面试记录样本
- `structured_interview.md`：已结构化面试记录样本

### 6.2 数据原则
- 测试数据与生产数据完全隔离
- 使用tmp_path确保测试后自动清理
- 禁止在测试中读写项目真实数据目录

## 7. 常见测试场景

### 7.1 异常路径测试
```python
def test_extract_empty_file_returns_empty_list(mock_llm, tmp_path):
    """空文件应返回空问题列表"""
    empty_file = tmp_path / "empty.md"
    empty_file.write_text("")

    result = extract_questions(str(empty_file))
    assert result == []
```

### 7.2 边界条件测试
- 空输入文件
- 超长文本输入
- 无效文件路径
- LLM返回格式异常
- 网络超时（Mock）

### 7.3 回归测试触发场景
- 修改检索算法 → 重跑question_bank测试
- 修改抽取逻辑 → 重跑extractor测试
- 修改配置结构 → 重跑config测试
- 修改API签名 → 重跑对应路由测试
