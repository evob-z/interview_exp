# 开发规范文档

> 版本：v1.0 | 最后更新：2026-05-17 | 状态：初稿

## 1. 代码风格规范

### 1.1 Python代码风格
- 遵循 PEP 8 规范
- 行宽限制：120字符
- 缩进：4空格（禁止Tab）
- 字符串：优先使用双引号 `""`，f-string格式化
- 类型标注：公开函数必须添加类型标注

### 1.2 命名规范

| 类型 | 风格 | 示例 |
|------|------|------|
| 模块/文件 | snake_case | `question_bank.py` |
| 类名 | PascalCase | `QuestionBank`, `ProfileManager` |
| 函数/方法 | snake_case | `detect_changes()`, `search()` |
| 常量 | UPPER_SNAKE_CASE | `INTERVIEW_REPO_PATH`, `GIT_ENABLED` |
| 私有方法 | _前缀 | `_parse_structured_questions()` |
| 变量 | snake_case | `session_id`, `question_list` |

### 1.3 导入顺序
```python
# 1. 标准库
import os
import json
from pathlib import Path

# 2. 第三方库
from fastapi import APIRouter, Depends
from openai import OpenAI

# 3. 项目内部模块
from config import INTERVIEW_REPO_PATH
from llm_client import chat_completion
from logger import get_logger
```

### 1.4 文件组织
- 每个模块文件顶部包含模块级docstring说明用途
- 常量定义在导入之后、类/函数之前
- 类方法按 `__init__` → 公开方法 → 私有方法 排列

## 2. 注释规范

### 2.1 模块注释
```python
"""
问题库检索引擎

提供多维度的面试题目检索能力，支持关键词搜索、
分类过滤和项目别名加权等功能。
"""
```

### 2.2 函数注释
```python
def search(self, query: str, top_k: int = 5, boost_categories: list = None) -> list:
    """
    搜索匹配的题目。

    Args:
        query: 搜索关键词
        top_k: 返回结果数量上限
        boost_categories: 需要加权提升的分类列表

    Returns:
        按相关度排序的匹配题目列表
    """
```

### 2.3 行内注释
- 仅在逻辑不直观时添加注释
- 注释语言：中文（面向项目实际使用者）
- 避免翻译代码的注释（如 `# 设置变量x为5`）

## 3. Git提交规范

### 3.1 提交信息格式
```
<类型>: <简要描述>

[可选的详细说明]
```

### 3.2 提交类型

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | `feat: 添加岗位预测题生成功能` |
| fix | Bug修复 | `fix: 修复题库检索重复问题` |
| refactor | 重构 | `refactor: 重构LLM客户端重试逻辑` |
| docs | 文档 | `docs: 更新API接口文档` |
| test | 测试 | `test: 补充archiver单元测试` |
| chore | 杂项 | `chore: 更新requirements.txt` |
| style | 格式 | `style: 统一代码缩进和空行` |

### 3.3 提交策略
- **分功能提交**：每个独立功能/修复一个commit
- **直接合main**：单人开发模式，直接在main分支提交
- **打Tag发布**：重要里程碑通过Git Tag标记版本号

### 3.4 分支策略（预留）
当前为单人开发，直接在main分支工作。若后续协作开发：
- `main`：稳定版本
- `dev`：开发分支
- `feature/*`：功能分支

## 4. 项目结构规范

### 4.1 新增功能模块
1. 核心逻辑放在项目根目录（如 `detector.py`）
2. API路由放在 `api/routes/` 目录
3. 知识库扩展放在 `knowledge/` 目录
4. Prompt模板放在 `prompts/` 目录
5. 测试文件放在 `tests/` 对应子目录

### 4.2 配置管理规范
- 所有可配置项通过 `.env` 管理
- 新增配置必须同步更新 `.env.example`
- 配置加载统一在 `config.py` 中完成
- 可选功能通过 `*_ENABLED` 开关控制

### 4.3 路径规范
- **禁止硬编码绝对路径**：全部通过config获取
- **路径必须英文**：INTERVIEW_REPO_PATH必须为英文路径
- **使用pathlib**：优先使用 `Path` 对象而非字符串拼接

## 5. 错误处理规范

### 5.1 异常处理原则
- 每个模块捕获自己能处理的异常
- 不能处理的异常向上传递
- LLM调用必须包含重试和降级逻辑
- API层统一异常格式返回

### 5.2 日志记录规范
```python
logger = get_logger(__name__)

# 关键操作记录INFO
logger.info(f"开始处理文件: {filename}")

# 异常记录ERROR + 堆栈
logger.error(f"LLM调用失败: {e}", exc_info=True)

# 调试信息用DEBUG
logger.debug(f"检索结果: {results}")
```

## 6. 依赖管理规范

### 6.1 添加新依赖
1. 安装到conda环境：`pip install <package>`
2. 更新 requirements.txt：`pip freeze > requirements.txt` 后手动精简
3. 记录选择理由（可在技术选型报告中补充）

### 6.2 版本策略
- requirements.txt 中不锁定小版本号（如 `fastapi` 而非 `fastapi==0.104.1`）
- 仅在遇到兼容性问题时锁定版本

## 7. Prompt工程规范

### 7.1 Prompt模板管理
- 所有Prompt模板存放在 `prompts/` 目录
- 使用Markdown格式，支持变量占位符
- 文件命名：`{场景}_system.md`

### 7.2 Prompt设计原则
- 角色定义清晰
- 输出格式明确约束
- 包含示例（few-shot）
- 防止项目信息混淆的约束条款
